# Claude Build Update — Hybrid Live Pipeline with Periodic Claude Code Review

Update the product and technical plan to use a **hybrid ingestion model** that balances:
- low ongoing cost
- near-live public updates
- strong editorial credibility
- periodic higher-quality review and correction

The public website should appear regularly updated and data-driven, but the system should distinguish between:
1. **continuous low-cost automated intake**
2. **periodic deeper review and correction passes**

This hybrid approach is now a core product requirement.

---

## 1. Updated Ingestion Philosophy

The website should not depend on expensive deep reasoning for every update.

Instead, implement two parallel layers:

### Layer A — Continuous Low-Cost Live Intake
A lightweight automated pipeline runs frequently and keeps the site looking current.

Its job is to:
- collect candidate articles from feeds and search sources
- extract metadata and article text where possible
- identify likely relevant items
- cluster likely duplicate stories
- perform first-pass categorization
- assign provisional attribution scores
- insert/update provisional public records
- update live counters and recent events

This layer should be:
- fast
- cheap
- mostly deterministic
- tolerant of imperfect classification
- clearly marked internally as provisional where relevant

### Layer B — Periodic Deep Review and Correction
A deeper review layer periodically revisits the dataset and improves quality.

Its job is to:
- search more deeply for missed events
- review uncertain classifications
- refine attribution strength
- adjust weighted percentages
- merge duplicates more accurately
- improve summaries
- fill in missing country/company/sector details
- promote provisional events into reviewed events
- revise prior totals if the evidence supports corrections

This second layer should act like an editorial correction and enrichment pass.

---

## 2. Public Data Freshness Model

The site should behave like a live tracker, but support two confidence levels of data:

### Provisional
Generated by the lightweight automated pipeline.
These records:
- can appear quickly
- can populate charts and news feeds
- can influence totals
- may have lower confidence
- may later be corrected

### Reviewed
Reviewed and corrected by the deeper classification process.
These records:
- carry higher confidence
- should be preferred in totals and featured news
- may replace or revise provisional figures
- should visibly improve trustworthiness

The system should support showing:
- last live update
- last reviewed update
- percentage of recent records reviewed
- count of provisional vs reviewed events

This should make the site feel active while remaining intellectually honest.

---

## 3. Updated Source Discovery Model

The low-cost live pipeline should continuously ingest from structured and semi-structured sources such as:

- RSS feeds
- major news source feeds
- business and finance feeds
- company press release pages
- newsroom pages
- investor relations pages
- layoff-related watchlists
- keyword-based search results
- known company pages
- historical source recrawls
- sitemap diffs where available

The system should not rely on deep reasoning to search the entire web every cycle.

Instead:
- gather broad candidate inputs cheaply
- run lightweight relevance screening
- pass only promising records to classification and storage
- reserve deeper review for periodic correction passes

---

## 4. Two-Tier Classification Model

### Tier 1 — Provisional Classification
Used by the live automated pipeline.

This stage should:
- quickly classify whether the article is likely relevant
- assign category:
  - AI layoffs
  - robot / automation layoffs
  - AI jobs created
  - productivity / GDP impact
  - unrelated
- assign provisional attribution:
  - explicit
  - strong
  - moderate
  - weak
  - fringe
  - not relevant
- estimate provisional weighted contribution
- create a short safe summary
- flag ambiguity

This tier can be somewhat aggressive in surfacing likely candidates, but it should include confidence scoring and uncertainty handling.

### Tier 2 — Reviewed Classification
Used during periodic deeper passes.

This stage should:
- revisit provisional records
- confirm or revise category
- revise weights and totals
- refine dedupe clustering
- detect missing related reporting
- improve public summaries
- upgrade confidence score
- mark the record as reviewed

Reviewed classifications must override provisional classifications when conflicts occur.

## 6. Public Presentation of Live vs Reviewed Data

The homepage, charts, and news pages should support this distinction without making the UX feel overly academic.

Recommended presentation:
- main totals use the best available data
- where a reviewed version exists, it overrides provisional values
- where only provisional data exists, it is included but flagged internally

Optional subtle UI elements:
- “Includes reviewed and provisional reports”
- “Recent records are being continuously verified”
- “Methodology explains how estimates are refined over time”

The news page can show badges such as:
- Provisional
- Reviewed
- Verified explicit
- Strong AI link
- Disputed
- Corrected

Avoid making the interface noisy, but ensure the data lifecycle is real and inspectable.

## 7. Updated Aggregation Rules

The aggregate system must support corrections over time.

### Rules:
- provisional events can contribute to totals immediately
- reviewed events replace provisional values for the same event
- merged duplicates must not double count
- if a later review lowers an event’s AI attribution, historical totals should update accordingly
- daily aggregate tables should preserve both:
  - snapshot totals as of that day
  - recalculated totals under current methodology

Store:
- `daily_total_conservative`
- `daily_total_core`
- `daily_total_upper`
- `reviewed_share_of_total`
- `provisional_share_of_total`

This enables both:
- live evolving totals
- trust-building transparency

- target duplicate clusters

This review layer should act as the quality-control engine for the site.

## 11. Data Pipeline Architecture Update

The ingestion architecture must now explicitly support:

### A. Discovery pipeline
- candidate source collection
- metadata fetch
- text extraction
- basic dedupe
- cheap relevance scoring

### B. Provisional classification pipeline
- first-pass categorization
- first-pass weighting
- provisional event creation
- live total updates

### C. Review pipeline
- deep re-analysis
- richer clustering
- weight correction
- evidence enrichment
- confidence revision
- reviewed event publishing

### D. Aggregate rebuild pipeline
- recalculate totals after any reviewed corrections
- update chart series
- update event counts
- update provisional/reviewed shares

This should be documented as a first-class system design, not an afterthought.

---

## 12. Cost-Minimization Requirement

The implementation should be optimized so that the public site does not require expensive full-depth reasoning for every update.

Design for:
- cheap frequent discovery
- cheap provisional classification where possible
- periodic heavier review
- minimal costly inference on the public-serving path
- clear separation of ingestion, review, and presentation

The site should remain operational even if the deep review layer is temporarily unavailable.

That means:
- live intake can continue independently
- reviewed corrections can be applied later
- frontend should gracefully handle mixed data quality states

## 13. Methodology Page Update

Update the methodology page to explain:

- that the tracker uses a live estimate model
- that some events enter provisionally and are refined later
- that reviewed records override initial estimates
- that totals may be revised as better evidence is found
- that the tracker prioritizes transparency over fake precision

This is a credibility feature, not just a legal note.

## 18. Final Product Principle

This tracker should feel both:
- **alive**
- **correctable**

It should feel current enough to attract attention every day, but rigorous enough that later deeper review improves the dataset instead of undermining it.

Design the system so that:
- fresh data can appear quickly
- stronger evidence can refine it later
- the public record becomes more accurate over time
- confidence and transparency increase as the system matures

This hybrid live-plus-reviewed model is now a core product requirement and should be reflected throughout:
- architecture
- schema
- pipelines
- homepage
- methodology
- admin tools
- charts
- filters
- status badges
- aggregate logic
